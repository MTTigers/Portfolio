你是一名资深产品经理与UX设计专家，具备从商业目标到用户体验的全链路分析能力。

我将提供一个界面（或多个界面组成的流程）。请你基于这些信息，完成以下分析：

### **第一部分：业务视角分析（必须优先完成）**

请从产品与商业角度出发，回答：

1 核心操作识别（Core Action）

明确这个页面/流程中：用户最核心的一步操作是什么（只能有一个）

要求：

- 必须是“对业务最关键”的行为，而不是表层点击行为
- 用一句话表达（动作 + 结果）

2 为什么它是业务核心（Why It Matters）

用2–3句话说明：

- 为什么这个操作是整个页面/流程存在的核心
- 它在整个业务链路中的位置（例如：转化 / 留存 / 变现 / 激活）

3 关键业务指标（Business Metrics）

用户完成该核心操作后，最关键的业务指标是什么（3–5个）

必须包含：

- 指标名称（如：转化率 / 完成率 / 客单价 / 留存 / 使用频次）
- 指标变化方向（↑ / ↓）
- 可选：影响范围（新用户 / 老用户 / 某类用户）

4 商业逻辑成立条件（Business Logic）

用通俗语言解释：

- 当这些指标提升时，意味着什么商业逻辑成立了？
（例如：用户需求被验证 / 转化路径有效 / 付费意愿成立 / 产品价值被感知）

5 总结：业务目标（必须输出一句话）

输出一条“可用于汇报或作品集”的总结：

格式参考：

「通过提升【核心操作】的完成效率与转化表现，驱动【关键指标】提升，从而验证【某个商业逻辑成立】。」

### 第二部分：用户体验问题分析（基于真实使用）

在明确业务目标后，请你：模拟用户真实使用该界面/流程，从用户视角出发，找出最关键的体验问题

核心要求（非常重要）

- 只输出 3–5个问题
- 必须按照 对业务指标影响权重排序（从高到低）
- 每个问题必须：
    - 明确影响了哪个业务指标
    - 说明“为什么会影响”（因果关系要讲清楚）

每个问题必须包含以下结构：

1 问题描述（What）：用一句话明确问题本质（不是表象）

2 具体表现（Where & Detail）

必须结合界面细节分析，例如：

- 信息架构（结构是否清晰）
- 交互路径（步骤是否多/绕）
- 视觉层级（主次是否明确）
- 视觉权重（是否引导行为）
- 比例与尺度（是否影响判断）
- 一致性 / 多样性（是否造成认知负担）

3 影响的业务指标（Impact）

明确影响：

- 哪个指标（如转化率 / 完成率 / 时长 / 跳出率）
- 如何影响（↑ 或 ↓）

4 原因解释（Why - 大白话）

用通俗语言解释：

- 为什么这个问题会导致指标变化
- 用户在这个环节“卡住 / 犹豫 / 放弃”的真实心理

输出风格必须遵守：

- 不复述输入内容
- 不做泛泛而谈
- 不讲“设计原则”，只讲“问题 → 影响 → 原因”
- 每一句都是结论
- 用通俗易懂但专业的表达
- 可直接用于汇报 / 作品集

第一部分：业务视角分析

1. 核心操作识别
用户选择符合模板要求的 CAD 映射属性文件并完成导入，让 CAD 属性映射规则进入系统生效。

2. 为什么它是业务核心
这个页面的核心是把外部 CAD 属性规则导入系统，形成后续自动识别、匹配和管理的基础配置。它处在业务配置和数据标准化环节，导入成功后，系统才能减少人工录入和属性维护成本，支撑后续图纸、物料、BOM 或文档管理流程。

3. 关键业务指标
CAD 映射属性导入完成率 ↑：配置用户 / 管理员
首次导入成功率 ↑：新用户 / 首次配置用户
单次导入耗时 ↓：高频配置用户
有效映射规则入库量 ↑：CAD / PLM 业务团队
后续 CAD 属性识别成功率 ↑：下游业务用户

4. 商业逻辑成立条件
当导入完成率和首次导入成功率提升，说明用户能够独立完成复杂配置，系统具备承接 CAD 标准化数据的能力。当有效映射规则入库量和后续识别成功率提升，说明导入的不是一份文件，而是能真正驱动业务自动化的规则资产。当单次导入耗时下降，说明配置路径足够清晰，用户不需要依赖实施或客服反复协助。

5. 业务目标总结
通过提升 CAD 映射属性文件成功导入的完成效率与转化表现，驱动导入完成率、首次导入成功率、有效映射规则入库量、CAD 属性识别成功率提升，从而验证系统能够低成本承接 CAD 属性标准化配置并支撑后续业务自动化。

第二部分：用户体验问题分析

问题 1：页面没有提前说明文件规则，导致用户上传前无法判断自己该选什么文件。
具体表现：文件区域只显示“请选择文件”和“浏览”，没有展示支持格式、文件大小、模板字段要求、示例文件名或导入前校验规则。“导入参考模板”是唯一规则入口，但它没有说明“必须按模板填写”，也没有和文件选择动作形成强绑定。
影响的业务指标：首次导入成功率 ↓；导入完成率 ↓；单次导入耗时 ↑。
原因解释：用户进入这里时并不知道系统要什么样的文件，只能凭经验选一个试试。第一次就失败的概率越高，用户越容易觉得这个功能难用，后续会转向线下问人或放弃配置。

问题 2：“数据预览”在未选择文件时仍作为可操作按钮出现，增加无效路径。
具体表现：底部按钮中“数据预览”与“取消”并列，视觉上是可点击的正常按钮。但当前状态没有任何文件，用户点击预览大概率无法得到有效结果，页面没有通过禁用态或提示说明预览的前置条件。
影响的业务指标：操作路径耗时 ↑；无效点击率 ↑；导入完成率 ↓。
原因解释：用户会把“数据预览”理解为下一步，尤其是担心导入错误时更想先预览。结果点了才发现不能用或没有内容，会让用户感觉流程不明确，信任感下降。

问题 3：核心按钮“确定”没有说明最终结果，用户不确定点击后是上传、校验还是导入。
具体表现：主按钮文案是“确定”，没有表达“确认导入”或“开始导入”。在导入类场景中，用户需要知道点击后是否会立即提交数据、是否还能预览、是否会覆盖已有映射规则，但界面没有给出任何提示。
影响的业务指标：导入完成率 ↓；操作犹豫时长 ↑；误操作率 ↑。
原因解释：CAD 映射属性属于配置数据，用户会担心导错影响后续业务。按钮越模糊，用户越不敢贸然点击，关键动作就会被拖慢。

问题 4：模板入口的重要性被低估，用户容易跳过标准化步骤。
具体表现：“模板下载”和“导入参考模板”只是普通文本加蓝色链接，视觉权重低于文件选择区和底部按钮。页面没有把“下载模板 → 填写模板 → 选择文件 → 预览 / 导入”组织成明确顺序，用户很容易直接从文件选择开始。
影响的业务指标：文件校验通过率 ↓；首次导入成功率 ↓；客服 / 实施支持成本 ↑。
原因解释：用户通常想最快完成任务，不会主动研究一个看起来像辅助说明的链接。如果模板是成功导入的前提，却没有被明显强调，用户就会先犯错，再返工。

问题 5：只能上传单个文件，限制了批量配置场景的效率。
具体表现：文件选择区只提供一个文件输入框，没有批量选择、拖拽上传、文件队列或多文件校验能力。对于 CAD 映射属性这类配置场景，用户可能需要一次性导入多个分类、多个项目或多个来源的映射文件，但当前流程只能重复“选择文件 → 预览 / 导入 → 再次打开弹窗”。
影响的业务指标：单次配置耗时 ↑；导入完成率 ↓；有效映射规则入库量 ↓。
原因解释：用户如果有多份文件要处理，就必须重复操作多次。次数一多，用户会觉得这个功能只适合小规模试用，不适合真实业务批量配置，容易转向线下处理或让实施人员代操作。

# Prompt CN

你是一名资深产品经理、交互设计专家与 UX 策略顾问，擅长从“问题—原因—业务影响”出发，基于原始界面或旧版流程截图，提出具有明确业务价值的改版方案，并将方案整理为可落地的 DRD 改版文档。

### 我将提供两类输入：

1. 界面输入
    - 单个页面截图，或多个页面组成的流程截图
    - 可以是原始版本、旧版本、现状版本，或问题版本
2. 问题输入
    - 问题描述
    - 具体表现
    - 影响的业务指标
    - 对问题原因的解释

你的任务是：针对每一个问题，输出至少 3 个具体、可执行、差异明显的交互设计改版方案，并在必要时整理为 DRD 改版文档结构。

### 输出目标

- 方案必须是可执行的，不是空泛建议
- 方案必须对业务目标有明确价值
- 方案必须能和改版前形成明显对比，让人一眼看出变化
- 方案必须体现交互设计与 UI 层面的具体改动
- 方案必须解释“为什么这样改更好”
- 方案必须说明会如何影响业务指标和业务目标

### 你的思考原则

你在生成方案时，要把以下“好体验”标准当作判断框架，而不是照搬结构输出：

- Usable（可用）：操作清晰、容易上手
- Equitable（公平）：适合不同背景、能力的人群
- Enjoyable（愉快）：使用过程具有更好的情绪体验
- Useful（有用）：真正解决用户问题

注意：这些标准不是输出标题，而是你判断方案优劣时的底层标准。

### 输出结构

请严格按照以下结构输出：

### 1. 问题定义

针对当前问题，先用简洁专业的话说明：

- 这个问题在界面或流程中的具体位置
- 它是如何发生的
- 它为什么会影响业务目标
- 它当前主要影响的业务指标是什么

### 2. 改版方案（每个问题至少输出 3 套）

2.1 每套方案都必须包含以下内容：

- 方案名称
- 用一句话概括方案核心方向
- 方案核心思路
- 说明这套方案是通过改变什么，来解决什么问题
- 具体改版设计

2.2 必须写具体，不能泛泛而谈。尽量覆盖以下维度：

- 信息架构调整
- 交互路径调整
- 操作顺序优化
- 视觉权重变化
- 信息层级重组
- 比例尺度变化
- 一致性与多样性的处理
- 关键按钮、模块、状态、反馈的改法
- 必要时补充文案、引导、默认值、确认机制、风险提示等

2.3 改版前后最明显的差异，要说清楚：

- 改版前用户看到的是什么
- 改版后用户首先感知到什么变化
- 哪些行为路径被缩短、强化或弱化
- 为什么别人一眼就能看出这是一次有效改版

2.4 为什么这个方案更好

从体验和业务两个层面解释：

- 为什么更可用、更有用，或更容易理解
- 为什么更符合用户真实决策过程
- 为什么它比原方案更有机会提升结果

2.5 对业务目标与业务指标的影响

明确说明这套方案会影响什么指标，例如：

- 转化率
- 点击率
- 完成率
- 使用时长
- 跳失率
- 中断率
- 复访率
- 留存率
- 客单价
- 发券领取率 / 使用率
- 下单率等

并解释：

- 它具体通过什么机制影响这些指标
- 最终对业务目标有什么价值

### 3. 三套方案对比总结

请用表格或分点方式总结三套方案的差异，包括：

- 适用场景
- 改动幅度
- 风险高低
- 业务收益方向
- 更适合快速上线，还是适合中长期优化

### 4. 输出 DRD 改版文档版本（可直接用于评审）

请把最终推荐方案整理成 DRD 结构，至少包含：

4.1 改版背景

- 当前问题
- 触发原因
- 对业务的影响

4.2 改版目标

- 要改善的核心体验问题
- 要影响的核心业务指标

4.3 设计策略

- 本次改版采取的核心思路
- 为什么这样设计

4.4 详细改版说明

按模块或流程节点逐条写清楚：

- 改什么
- 为什么改
- 用户感知变化是什么
- 对业务的作用是什么

4.5 风险与验证建议

- 可能带来的副作用
- 上线后建议观察的指标
- 可用什么方式验证效果（如 AB 测试、可用性测试、漏斗对比等）

### 输出要求

- 必须结合我提供的截图和问题描述来分析，不能脱离界面空讲
- 必须说人话，专业但容易理解
- 不要只讲“体验更好了”，必须讲清楚为什么更好
- 不要只讲“优化建议”，必须讲成可执行的设计方案

03 Component systemization

让设计调整进入源码和真实页面

在原型验证之后，我把高频组件抽象到 React 组件和 token 控制台。通过控制台调整颜色、圆角、间距、状态和尺寸，组件源码会同步变化，具体 React 页面也会跟随更新。

Token lab

组件 Token 定制

组件 token 包括组件自己的局部 token，也包括全局共用 token。通过调整颜色、圆角、间距、密度和状态变量，可以快速得到一套定制化组件库。

局部 token：控制 Button、Dropdown、Input、Table 的组件特性 全局 token：统一品牌色、圆角、间距、字体和状态反馈 输出结果：组件库从默认样式变成可复用的定制设计系统
Button token 按钮颜色、尺寸、圆角和状态变量调整。
Dropdown token 下拉菜单层级、选项状态和交互反馈调整。
Input token 输入框边框、聚焦、禁用和校验状态调整。
Table token 表格密度、表头、行高、分割线和状态样式调整。

Workflow principle

AI is not the output. It is the workflow.

我的价值不在于让 AI 自动产出一个看起来完整的结果，而在于设计一套可判断、可验证、可沉淀的工作流。AI 负责扩展分析与生成空间，设计师负责识别问题、校准体验、控制风险，并把结果推进到可复用系统。
